PokeDex in Action



    GitHub Working with the PokeAPI

    The first step to building the PokeDex was to create 4 functions for the "Network Engine" class that would handle the API call for the different endpoints (see Github for NetworkEngine class). The PokeAPI's JSON response has the URLs for different end points nested in the properties.

    The class functions take in an @escaping closure as a parameter and uses a "Result" type. The purpose for @escaping is to inform the caller of the function that takes a closure that the closure might be stored or outlive the scope of the receiving function. This means when making a data call using the network engine methods that the closure must take care of retain cycles by declaring a weak reference to the class. The calling function in this case "fetchData()" in the mainVC uses the network engine methods to call for the data from the PokeAPI and continues when a response is received.


    GitHub Building the UI

    I built the PokeDex using the MVVM design pattern and designed each view controller programmatically. The images on the left show the "PokemonDetailsViewController" which contains a table view with details about the Pokemon. The data for the table starts from the "fetchData()" function (line 173 in Github) which makes the API call using the "Network Engine" methods and returns a response. If successful the responses will contain the data for the "PokemonSpeciesDetails" and "PokemonEvolutionDetails" which are both model objects.

    Registering custom table view cells

    Once the API call has completed, the table view is registered with its custom table cell views (line 141 in Github). Both the custom table cells (PokemonFlavorTextTableCell and PokemonEvolutionsTableCell) have a view model property that is set in the "cellForRowAt" function (line 303 in Github).

    Setting up the Pokemon fact cell

    The "PokemonFlavorTextViewModel" is initialized with the "pokemonSpeciesDetails" object (line 316 in Github) and sets up the different facts about the pokemon. The "PokemonFlavorTextViewModel"s init function modifies the fact array to only show facts written in english. It also has a "mutating" method that helps cycle the different facts in the array. This method is called from the "PokemonFlavorTextTableViewCell" (line 68) and uses the protocol-delegate pattern with the "PokemonDetailsVC" to detect the button tap and propagate a new fact up to where the protocol method was implemented.

    Setting up the Pokemon evolution cell

    Setting up the evolution chain required traversing the PokemonEvolution objects chain property using recursion. The outcome is to fill an array of PokemonEvolutionHierarchy objects that contain the name and evolution level for each pokemon in the chain.

    The way this works is by calling “getEvolutionNames(evolutionChain: )” method passing in the PokemonEvolutions.Chain property (line 207 in Github). This property contains the evolution chain for the particular pokemon and is the starting point for extracting the name and level.

    The first pokemon in the chain is added to the evolution array. and a check on the property “evolves_to” sees if the array is not empty which will trigger the recursion and call the “traverseChain(chainLink: )” method passing in the PokemonEvolutions.ChainLink array (line 216 in Github).

    The traverseChain method goes through the chainlink creating a PokemonHierarchy object of a name and evolution level and adding the object to the evolutions array. The recursion stops once the chainlink property is empty meaning there is no more chain to follow.

    The result is an array containing objects of a pokemon name and evolution level that can be used for the Evolution section of the tableview. The pokemonEvolutions array is also used to retrieve the sprite images. Looping over the array and matching the name property in the main pokemonList allows for getting the image from the URL. With the PokemonEvolutions and pokemonImages arrays set, the table view section for the evolution chain can now be created using the MVVM pattern to initialize the cell with a “PokemonEvolutionsViewModel” using the arrays (line 327 in Github).
